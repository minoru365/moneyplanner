## 1. 整合性（同じ操作・概念が画面ごとに違う）

- [x] **A3 / 中** — 履歴の検索条件がカレンダー表示に切り替えても残り、リストとカレンダーで同じ日のヒット件数が食い違う。`selectedDateTxs` も `transactions`（フィルタ結果）と `calendarTransactions` の両方を見ている。

- [x] **A4 / 低** — 集計→履歴ドリルダウンで「合計」行をタップした場合、種別と期間のみが条件としてつくが、検索パネルのサマリーから「集計から来た」ことが分からない。

## 2. データ損失／壊しそう

- [x] **B1 / 高** — カテゴリ削除確認文言が「『name』を削除しますか？」のみ。実際は過去取引の `categoryId` が null になる（未分類化）が伝わらない。件数も影響範囲も明示なし。
  - 該当: `app/(tabs)/settings.tsx:532-566` / `lib/firestore.ts:901-936`
  - 推奨: 影響を受ける取引件数を表示、「過去◯件の取引はカテゴリ未設定になります」

- [x] **B2 / 高** — カテゴリ/内訳リセットは `budgets / stores / storeCategoryUsage / breakdowns / categories` を全削除＋再投入＋取引リンク貼り直しを逐次実行。途中失敗で半壊状態のまま家族側にも同期される可能性。
  - 該当: `lib/firestore.ts:728-843`
  - 推奨: `runTransaction` または batch化、失敗時の自動リトライと明示エラー

- [x] **B3 / 中** — 口座削除時に手入力された初期残高分が引き継がれず消える。確認文言は正しいが、ユーザーから見て「お金が消えた」と感じやすい。
  - 該当: `app/(tabs)/settings.tsx:441-464` / `lib/firestore.ts:1437-1495`
  - 推奨: 「初期残高として手動設定された ¥X は引き継がれません」と数値で警告

- [x] **B4 / 高** — `updateAccountBalance` が `initialBalance = balance - transactionNet` で常に逆算上書き。家族同時操作で初期残高が誤書換、以降の残高表示がずれ続ける。
  - 該当: `lib/firestore.ts:1372-1392` / `app/(tabs)/settings.tsx:418-423`
  - 推奨: トランザクション内で取引合計を取得し直す、または「初期残高」直接編集UIへ

- [x] **B5 / 中** — 招待コード再発行のレース条件。古いコードで参加処理中に再発行されても、家族側 `joinHousehold` が中途で書き込みを継続する可能性。
  - 該当: `lib/household.ts:151-183,63-99`
  - 推奨: invite コードドキュメントに有効期限／参加側で世帯側 `inviteCode` を再確認

- [x] **B6 / 高** — 「認証解除と全データ削除」確認文言が薄い。世帯メンバーがいても警告なく全員のデータ消える。
  - 該当: `app/(tabs)/settings.tsx:789-823` / `lib/firestore.ts:703-726`
  - 推奨: メンバー2人以上のとき「他の家族（◯名）も使えなくなります」を太字、世帯から退出を先に勧める導線

## 8. iOS固有

- [x] **H1 / 中** — 履歴の月ピッカーや検索パネルの日付ピッカーが inline `spinner` で、モーダルでない。背後ScrollViewを操作してもピッカーが残ったままになる。
  - 該当: `app/(tabs)/history.tsx:943-986` / `components/HistorySearchPanel.tsx:376-430`
  - 対応: 履歴ScrollViewのスクロール開始時に月ピッカーと検索日付ピッカーを自動dismiss

- [x] **H2 / 中** — `TransactionEditor` のメモTextInputが `multiline + maxLength=100`。改行送信時の `blurOnSubmit` がiOSで想定通り動かない場合、キーボード閉じない。
  - 該当: `components/TransactionEditor.tsx:355-372`
  - 対応: iOSでメモ入力フォーカス中に専用の「メモ入力を完了」ボタンを表示し、キーボードを明示的に閉じられるようにした

- [x] **H3 / 低** — 設定タブのカテゴリ管理モーダル＋editor＋numericInputModalの入れ子が複雑。画面回転やiPad split-viewで崩れる可能性。
  - 該当: `app/(tabs)/settings.tsx:1156-1939`
  - 対応: numeric入力モーダルを親モーダル外へ移動して入れ子を浅くし、キーボード/数値入力の状態を閉じる共通処理で遷移を整理
  - **方針転換（2026-06-12）**: 上記の「親モーダル外へ移動」は退行を起こしていた。numeric入力は `useNativeModal={false}`（インライン描画）のため、ネイティブModalである管理モーダルの背面に描画され、予算・初期残高のテンキーが一切表示されない（＝入力不能）状態だった。インライン要素はネイティブModalより前面に出せないため、numeric入力を管理モーダルの内側末尾（編集オーバーレイより前面）へ戻した。入れ子の浅さより入力可能性を優先する。今後この入れ子を再度解消する場合は、numeric入力を `useNativeModal={true}`（ネイティブModal同士のスタック）にするか、管理モーダル自体の非Modal化とセットで行うこと
  - 追記（2026-06-12）: 同系統のレイヤー問題が履歴編集の金額テンキーにもあった（`amountInputUseNativeModal={false}` のインライン描画が編集モーダルのヘッダーより背面になる）。日付/カテゴリ/口座/店舗ピッカーと同じネイティブModal方式に統一し、`amountInputUseNativeModal` プロパティ自体を撤去。**金額・数値入力系のインライン描画（`useNativeModal={false}`）は、ネイティブModalと重なる画面では使わないこと**
