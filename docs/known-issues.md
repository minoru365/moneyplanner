# 既知の懸念事項リスト

実機で家族2人が日常的に使ったときに「困る／怒る／壊れる」可能性のある箇所を、機能実装一通り完了後（2026-05-08時点）にレビューして洗い出したリスト。

build 14-21の実機検証で既に挙がっている既知課題は除外している。`docs/testflight-history.md` の発見事項も併せて参照する。

未実装課題は対応時にチェックを入れ、対応buildを記載する。新たな懸念は随時追記する。

## 次にやるべき改善（優先度トップ3）

1. **【高 / B1, B2】 カテゴリ削除・リセットの確認文言と原子性強化**
   「カテゴリ削除＝過去取引の categoryId が null になる（未分類化）」「リセットは予算引継ぎがあるが独自カテゴリは消える」を、件数つきで明文化。リセットは1トランザクションに近づけ、途中失敗時の中間状態を作らないように改修。

2. **【高 / B4】 口座残高編集の整合性修正**
   `updateAccountBalance` の `initialBalance = balance - transactionNet` 逆算は、家族間の同時操作と直交していない。`runTransaction` 内で取引合計を再計算するか、画面上は「初期残高」を直接編集するUIに切り替えて last-write-wins の被害範囲を限定する。同時にB3（口座削除時に消える初期残高分）の警告強化もセットで実施。

## 1. 整合性（同じ操作・概念が画面ごとに違う）

- [ ] **A2 / 中** — 設定タブの保存ボタンは `await` のままで、記録タブのような `WRITE_ACK_TIMEOUT_MS=900ms` で抜ける挙動と不揃い。オフライン時に画面ごとに反応が違う。
  - 該当: `app/(tabs)/index.tsx:52,284-343` / `app/(tabs)/settings.tsx:478-518,403-428`
  - 推奨: 設定保存も `waitForPendingWrite` 経由で揃える、または「同期待ち」トースト

- [ ] **A3 / 中** — 履歴の検索条件がカレンダー表示に切り替えても残り、リストとカレンダーで同じ日のヒット件数が食い違う。`selectedDateTxs` も `transactions`（フィルタ結果）と `calendarTransactions` の両方を見ている。
  - 該当: `app/(tabs)/history.tsx:225-345`
  - 推奨: カレンダー表示時は検索条件を非適用にする

- [ ] **A4 / 低** — 集計→履歴ドリルダウンで「合計」行をタップした場合、種別と期間のみが条件としてつくが、検索パネルのサマリーから「集計から来た」ことが分からない。
  - 該当: `app/(tabs)/summary.tsx:561-577,620-643` / `lib/historyDrilldown.ts`
  - 推奨: サマリーラベルに「（集計から）」など付与

## 2. データ損失／壊しそう

- [ ] **B1 / 高** — カテゴリ削除確認文言が「『name』を削除しますか？」のみ。実際は過去取引の `categoryId` が null になる（未分類化）が伝わらない。件数も影響範囲も明示なし。
  - 該当: `app/(tabs)/settings.tsx:532-566` / `lib/firestore.ts:901-936`
  - 推奨: 影響を受ける取引件数を表示、「過去◯件の取引はカテゴリ未設定になります」

- [ ] **B2 / 高** — カテゴリ/内訳リセットは `budgets / stores / storeCategoryUsage / breakdowns / categories` を全削除＋再投入＋取引リンク貼り直しを逐次実行。途中失敗で半壊状態のまま家族側にも同期される可能性。
  - 該当: `lib/firestore.ts:728-843`
  - 推奨: `runTransaction` または batch化、失敗時の自動リトライと明示エラー

- [ ] **B3 / 中** — 口座削除時に手入力された初期残高分が引き継がれず消える。確認文言は正しいが、ユーザーから見て「お金が消えた」と感じやすい。
  - 該当: `app/(tabs)/settings.tsx:441-464` / `lib/firestore.ts:1437-1495`
  - 推奨: 「初期残高として手動設定された ¥X は引き継がれません」と数値で警告

- [ ] **B4 / 高** — `updateAccountBalance` が `initialBalance = balance - transactionNet` で常に逆算上書き。家族同時操作で初期残高が誤書換、以降の残高表示がずれ続ける。
  - 該当: `lib/firestore.ts:1372-1392` / `app/(tabs)/settings.tsx:418-423`
  - 推奨: トランザクション内で取引合計を取得し直す、または「初期残高」直接編集UIへ

- [ ] **B5 / 中** — 招待コード再発行のレース条件。古いコードで参加処理中に再発行されても、家族側 `joinHousehold` が中途で書き込みを継続する可能性。
  - 該当: `lib/household.ts:151-183,63-99`
  - 推奨: invite コードドキュメントに有効期限／参加側で世帯側 `inviteCode` を再確認

- [ ] **B6 / 高** — 「認証解除と全データ削除」確認文言が薄い。世帯メンバーがいても警告なく全員のデータ消える。
  - 該当: `app/(tabs)/settings.tsx:789-823` / `lib/firestore.ts:703-726`
  - 推奨: メンバー2人以上のとき「他の家族（◯名）も使えなくなります」を太字、世帯から退出を先に勧める導線

## 3. オフライン／同期の罠

## 5. エラー表示／フィードバック

- [ ] **E1 / 中** — `handleSaveCategory` で予算入力が不正計算式の場合、カテゴリ自体は既に save 済みでモーダル開いたまま `Alert.alert("エラー")` のみ。重複カテゴリ作成事故が起きうる。

## 6. 計算系の正確性

  - 該当: `app/(tabs)/history.tsx:201-213` / `lib/firestore.ts:1700-1717`
  - 推奨: `fromYearMonthDate / toYearMonthDate` を関数化、月境界（うるう年含む）テスト

## 7. 権限・世帯まわり

- [ ] **G1 / 中** — メンバー解除は `removedAt` の softdelete のみ。同じユーザーが招待コードで再参加すると `removedAt` が消えて復活。
  - 該当: `lib/household.ts:205-220,63-99`
  - 推奨: 解除時に再参加を許可しないフラグ／招待コード再発行を促す

  - 該当: `app/(tabs)/settings.tsx:869-897`
