# アカウント削除のApp Store対応（2026-07-15）

## 背景

バージョン1.0（build 34）がApp Store審査で **Guideline 5.1.1(v)（アカウント削除必須）** により却下された。アプリには設定画面の「認証解除と全データ削除」として削除機能が実装済みだったが、以下の構造により審査員が到達できなかったと推定した。

- Apple Sign-In直後の世帯未参加ユーザーは世帯設定画面（`app/household.tsx`）に着地するが、この画面にはアカウント削除もログアウトも存在しなかった。設定タブは世帯参加後にしか開けない
- 設定画面でも、削除ボタンは既定で閉じている「アカウント」折りたたみセクション内にあり、文言も「認証解除と全データ削除」で"アカウント削除"と読み取りにくかった

また、Sign in with Apple併用アプリのアカウント削除ではAppleトークンの失効（revoke）がApple要件だが、従来実装はFirebase Authの `deleteUser` のみでトークン失効を行っていなかった。

## 決定

1. **世帯未参加でもアカウント削除に到達できる導線を追加する**
   - 世帯設定画面（選択モード）に「ログアウト」「アカウントを削除」リンクを追加
   - 削除フロー: 確認ダイアログ → Apple再認証＋トークン失効 → `users/{uid}` 削除（承認待ち参加リクエストがあれば先に取り下げ）→ Firebase Authアカウント削除
   - 世帯参加中ユーザーがこの導線に入った場合はエラーにして設定画面の削除フローへ誘導する（`deleteCurrentUserProfileWithoutHousehold` のガード。世帯データの孤児化防止）
2. **設定画面の削除導線を発見しやすくする**
   - ボタン文言を「認証解除と全データ削除」→「アカウントを削除（全データ削除）」へ変更
   - 「アカウント」セクションを既定で展開に変更
   - 確認入力（「全データ削除」と入力）は誤操作防止としてApple許容範囲のため維持する
3. **Apple Sign-Inトークンの失効を削除フローに組み込む**
   - 再認証時に取得した `authorizationCode` で `revokeToken` を実行する（`reauthenticateAndRevokeAppleToken`）
   - modular APIの `revokeAccessToken` はReact Native Firebaseのネイティブで未実装のため、namespacedの `revokeToken` を使う
   - **失効はbest effortとし、失敗してもアカウント削除は続行する**。authorizationCodeは有効期限5分・使い捨てで確実な再試行手段がなく、削除完了（ユーザーの権利・5.1.1(v)の本義）を優先する
   - `auth/requires-recent-login` での再試行時（`deleteCurrentUserAccountWithReauthRetry`）は、再認証で得た新しいcodeで再度失効を試みる

## 理由

- 却下文面は「削除オプションが存在しない」であり、録画提出だけで返信しても、審査員が同じ経路（世帯未参加）をたどれば再却下されるため、導線の追加が必要
- トークン失効を再認証直後（データ削除前）に行うのは、大量データ削除に時間がかかるとcodeが失効するため。失効成功後にデータ削除が失敗しても、ユーザーは再度Apple Sign-Inで再認証して削除をやり直せるため復旧可能

## 再提出時の手順

- 実機で「サインイン → 削除導線へ移動 → 削除完了」の画面録画を撮影し、App Reviewへの返信とApp Review Informationのメモ欄に添付する（`docs/release-checklist.md` 参照）
