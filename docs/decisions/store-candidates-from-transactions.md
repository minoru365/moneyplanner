# お店候補を取引スナップショットから生成する（2026-07-04）

## 背景

お店選択の候補は `stores` と `storeCategoryUsage` から生成していたが、履歴検索の網羅性対応で
取引のローカルキャッシュを広く活用するようになった。取引にはすでに `storeNameSnapshot` と日付があり、
候補表示・検索・CSV往復に必要な情報は取引側だけで足りる。

一方、全履歴を一度読んだ後に検索窓やお店選択を開くと、全店候補の構築が重くなりやすい。候補を
「ローカル取引キャッシュから集約し、上位40件を検索語で絞る」方式に寄せると、追加のFirestore読み取りと
store masterの整合性管理を避けられる。

## 決定

- 通常の記録保存、履歴編集保存、選択コピー、CSV取り込みでは `storeId: null` を保存する
- 店舗名の表示・検索・CSV出力/再取り込みは `storeNameSnapshot` を正とする
- お店選択候補はローカルにキャッシュ済みの取引から店名を重複排除し、選択カテゴリで最近使った店を優先する
- 候補表示は上位40件に制限し、検索語がある場合は部分一致で絞り込んでから上限を適用する
- 履歴検索パネルのカテゴリ/内訳/お店候補も、現在表示中の `listTransactions` ではなくSDKローカルキャッシュ内の取引から生成する。`listTransactions` は履歴リスト本体と適用済み検索結果の表示母体として残す
- `stores` と `storeCategoryUsage` の関数・コレクションは旧データ互換のため当面残すが、通常導線では参照・更新しない

## 影響

- 店舗候補の網羅性は、その端末のFirestore SDKローカルキャッシュにある取引に依存する
- 履歴検索などで全履歴を取得した後は候補も自然に増える
- store masterの復元やカテゴリ別使用履歴の二重管理が不要になる
- 既存取引に残る `storeId` は表示上の正ではなく、店名スナップショットを優先する

## 代替案（不採用）

- **専用のstoreNameキャッシュをアプリ側に追加する**: Firestore SDKキャッシュと重複し、失効管理が増える
- **storesコレクションを継続して正にする**: CSV取り込み・カテゴリリセット・編集復元で整合性管理が増え、
  店名スナップショットを持つ現行データ構造と重複する
