# 集計・履歴の読み取りキャッシュ計画（案・未実装）

> 状態: **計画のみ（コード未改修）**。2026-06-14 に方針議論。実装は後日。
> 目的: 年切替/タブ往復のたびに発生する Firestore 読み取り（1年≈2,000件）を最小化し、
> アプリ再起動後もキャッシュを活かしつつ、家族の更新は取りこぼさない。

## 背景・課題

- 集計タブの取引購読は年単位（`yearScope`）。年を切り替えるたびに onSnapshot を張り直し、
  **その年の約2,000件をサーバーから読み直す（課金される）**。同じ年に戻る/短時間に複数年をめくると、
  そのたびに再読み込みされ積み上がる。
- 履歴も、フォーカスのたびに先頭ページ（約100件）を `.get()` し直している。

## 土台となる事実（要・実機確認）

- React Native Firebase の Firestore は**オフライン永続化がデフォルトON**。読んだデータはディスクに永続化され、
  **アプリ再起動後も残る**。`query.get({ source: 'cache' })` は**サーバーに行かず＝課金ゼロ・即時**で返る。
  → 「メモリキャッシュをアプリ終了後も保持したい」は、**自前ストレージ不要**で Firestore のディスクキャッシュを使えば実現できる。

## 設計：3層リード + 軽量フレッシュネスチェック

### 読み取り戦略（スコープ単位＝集計は「年」、履歴は「ページ」）
1. **セッションのメモリキャッシュ**（`Map<scopeKey, data>`）: あれば即描画。読み取り0・パース0。
2. **Firestore ディスクキャッシュ**（`get({ source: 'cache' })`）: メモリミス時。**課金0・再起動後も有効**。結果をメモリへ載せる。
3. **サーバー**（`get({ source: 'server' })`）: 「キャッシュが空」または「フレッシュネスチェックで“変更あり”」のときだけ。

### スコープの定義（キャッシュキー単位）
- **集計** = 年（`yearScope`）。月送り（同年内）はスコープ変更ではない。
- **履歴** = 日付範囲フィルタ（from/to）。種別/カテゴリ/内訳/店舗/メモの絞り込みは読み込み済みデータへの
  クライアント側フィルタ＝スコープ変更ではない。load-more（次ページ）もスコープ変更ではなく未取得分の取得。

### フレッシュネスチェック（タブフォーカス時／スコープ変更時）
家族や他端末の更新があるかを**安価に判定**してから、必要時だけサーバー全件読みする。
- マーカーのサーバー読みは**タブフォーカス時に1回**だけ行い「現在の版」をセッションで保持。
- **スコープ変更（年切替・日付範囲変更）時はマーカーを読み直さず**、保持中の「現在の版」と対象スコープの
  キャッシュ版を突き合わせて cache/server を判断（＝短時間の年パラパラ送りでマーカー読みが増えない）。
- ただし**自分のCRUD直後**はマーカーが上がるので、そのスコープは次表示時にサーバー再読み。

- **採用案：データバージョン・マーカー方式**
  - 小さなマーカーDocを用意（例 `households/{id}/meta/dataVersion`）。フィールドは `updatedAt: serverTimestamp` と任意で `count`。
  - **取引の add / update / delete のたびに、既存のバッチへマーカー更新を1件足す**（＋1 write/操作。単一Docなので軽い）。
    - **CSVインポートはper-rowで叩かず、バルク完了後に1回だけ**バンプ（重要：1万件分の書き込みにしない）。
  - フォーカス時の判定（AsyncStorage不要・Firestoreのキャッシュだけで完結）:
    1. マーカーを `source:'cache'`（無料）で読む＝前回同期時点の版
    2. マーカーを `source:'server'`（1 read）で読む＝現在の版
    3. 一致 → ディスクキャッシュ/メモリのスコープをそのまま表示（**サーバー全件読みしない**）
       不一致 → そのスコープをサーバー再読込し、メモリ更新
  - 典型コスト: 変更なしのフォーカス = **マーカー1 read のみ**（従来 約2,000 → 1）。変更ありのみ +約2,000。

- **代替案（参考）**
  - F2: マーカーを**年ごと**に持つ → 見ている年が変わった時だけ再読込（横断更新での無駄読みを排除）。書き込み側で
    `year(date)` を bump、日付が別年へ動く編集は新旧2年を bump。精度↑・実装やや複雑。
  - F3: マーカーDocを作らず、`count()`（1 read）＋ `orderBy('updatedAt').limit(1)`（1 read）で add/delete/edit を検知。
    書き込み増なし。ただしスコープ付き updatedAt 並べ替えに**複合インデックス（date+updatedAt）**が要る。

## 既存の仕組みとの整合
- 永続化は RN 既定でON（要・実機確認。明示設定は現状なし）。
- マーカーDocは世帯サブコレクション配下 → 既存の Rules（`allow read, write: if activeMember`）でカバー。**Rules変更不要**。
- 履歴は既にページング＋フォーカス更新。**フォーカス更新をマーカー判定でゲート**すれば、変更なし時の約100 read を1 readに。
- 集計は現在ライブ購読 → **キャッシュ＋マーカー方式へ置換**（ライブ購読はやめる）。

## トレードオフ（受け入れる挙動）
- 集計・履歴は**リアルタイムではなくなる**。家族更新は**タブ切替（フォーカス）時に反映**（=ユーザー方針どおり）。
  見ている最中の家族編集は、離れて戻る/プルリフレッシュまで見えない。
- マーカーDocは単一Docの書き込みホットスポット（Firestoreの単一Doc ~1 write/sec 目安）。世帯利用は低頻度で問題なし。
  バルクインポートは1回だけバンプ（per-row厳禁）。
- 採用案（世帯単位マーカー）は「別の年が変わった時でも、見ている年を一応再読込」する粗さがある（たまに約2,000の無駄読み）。
  気になれば F2（年別マーカー）へ。

## 想定変更範囲（後日実装時）
- 新規ユーティリティ/フック: キャッシュ＋フレッシュネス判定つきの取引リーダー（集計が使用。履歴はフォーカス更新の
  ゲートに同判定を流用）。
- `app/(tabs)/summary.tsx`: ライブ `useCollection` を上記リーダーへ置換（メモリ→ディスクキャッシュ→（必要時）サーバー）。
- `app/(tabs)/history.tsx`: フォーカス時の `refresh()` をマーカー判定でゲート（変更なしなら再読込しない）。
- `lib/firestore.ts`: add/update/delete とインポートでマーカーをバンプ。マーカー読み出しヘルパー（cache/server比較）。
- テスト: マーカーのバンプ条件、キャッシュ/再読込の判定を純関数化して検証。複合インデックスは採用案では不要。

## 確定事項（2026-06-14 承認済み・この方針で実装する）
1. マーカー粒度: **世帯単位**（`households/{id}/meta/dataVersion` を1件）。横断更新時の粗さは許容。
2. ライブ購読: **残さない**。集計・履歴ともフォーカス（タブ切替）時のフレッシュネスチェックのみで更新。
3. フレッシュネス方式: **マーカーDoc**（`updatedAt: serverTimestamp` をCRUDでバンプ、`cache`版と`server`版を読み比べ）。複合インデックス不要。
4. 適用範囲: **集計＋履歴のフォーカス更新ゲートを同時に**実装（判定ロジックを共通化）。
