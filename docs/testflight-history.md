## 記録ルール

- build番号ごとに確認結果を追記する

### build 26 発見事項（未対応。修正はテスト完了後にまとめて）

1. **世帯退出後、同じ世帯に再参加できない**
   - 再現手順: 世帯から退出（自分をメンバー解除）→ 同じ世帯の招待コードで再参加を試みる
   - 実際の結果: 「すでにこの世帯に参加しています」の通知が出て参加できない
   - 期待結果: 退出済みなら再度参加リクエストを送れる（または明確な理由の案内が出る）
   - 調査メモ（未検証の仮説）: 退出時に `users/{uid}.householdId` が即時クリアされず残るため、`requestJoinHousehold` の「すでに参加済み」判定に引っかかる可能性。また自己退出でも `members/{uid}.rejoinDisabled: true` が立つため、仮に判定を通過しても承認段階で「再参加できない状態」として弾かれる設計になっている点も要整理（自己退出と強制解除で再参加可否を分けるか要判断）

2. **最後の1人が世帯から退出できない**
   - 再現手順: 世帯のアクティブメンバーが自分1人の状態で、設定画面から自分を解除（退出）する
   - 実際の結果: 退出できない
   - 期待結果: 確定仕様どおり「最後のメンバー退出時は世帯データごと削除」される
   - 調査メモ（未検証の仮説}: 該当経路は `removeHouseholdMember` の最終メンバー分岐（全サブコレクション削除 → inviteCodes削除 → 世帯ドキュメント削除 → users更新）。①Security Rulesで退出直前/途中の削除が拒否されている、②2026-07-03のFirebase v22モジュラー移行がこの分岐を書き換えており（`getDocs(collection(householdRef, name))` / `deleteDoc(householdRef)` 等）移行リグレッションの可能性、の両面を確認する。エラーメッセージ/ログの採取が要る

3. **「認証解除と全データ削除」が permission-denied で失敗する**
   - 再現手順: 設定画面から「認証解除と全データ削除」を実行（確認入力・Apple再認証を通過後）
   - 実際の結果: エラー表示「[firestore/permission-denied] The caller does not have permission to execute the specified operation.」
   - 期待結果: 世帯配下の全データ削除 + `users/{uid}` 削除 + Authアカウント削除が完了する
   - 追加情報（2026-07-04）: エラー表示後にスワイプでリフレッシュすると世帯設定（作成/参加）画面に戻っており、**世帯データの削除自体は完了している模様**
   - 調査メモ（更新）: 削除処理の本体ではなく**後段で拒否が起きている**可能性が高い。有力仮説は ①世帯データ削除後もアクティブなFirestoreリスナー（設定画面の購読）が permission-denied を受け、それがエラー表示として出ている（削除成功時の想定挙動に近いが、ユーザーにエラーとして見えるのは問題） ②削除シーケンス終盤のステップ（`users/{uid}` 削除など）が members 削除後のRules判定変化で拒否される。#2（最後の1人退出不可）と同根かは要再検証 — #2も「エラー表示はあるが実は削除済み」の可能性がある
   - **確定（2026-07-04）**: Firebase Console の Authentication で実行後もユーザーが2人とも残存していることを確認。**permission-denied でフローが中断し、最終ステップの Firebase Auth アカウント削除が実行されていない**。「認証解除と全データ削除」の実態が「データ削除のみ・認証は残存」になっており仕様不一致。修正時は ①後段の permission-denied の発生源特定（リスナー or users削除） ②エラー発生時もAuth削除まで完遂するか、途中失敗時のリカバリ方針の整理、の両方が必要

4. **無効なQRを写しても警告文が表示されず無反応**
   - 再現手順: 参加画面「QRコードを読み取る」で招待コード以外のQR（URL等）をカメラに写す
   - 実際の結果: 何も表示されず無反応（読み取り自体を無視しているように見える）
   - 期待結果: 「招待コードのQRコードではありません」の警告文が約2秒表示される
   - 影響度: 低（誤参加は起きない。ただし無効QRを写したユーザーが「壊れている」と誤解しうる）
   - 調査メモ（未検証の仮説）: `InviteQrScanner` の `onBarcodeScanned` が発火していないか、発火しても `invalidMessage` の表示が機能していない。expo-cameraはスキャンイベントを高頻度連射するため、警告表示→即上書きクリアのタイミング問題の可能性もある。カメラが実際にQRを検出しているか（ログ）から切り分ける

5. **App Check「未確認: 無効なリクエスト」が71%（7日間で3,020/4,232件）**
   - 観測: 2026-07-04時点の過去7日間指標。確認済み29%・無効71%・古いクライアント0・送信元不明0。無効分の発生時期は7/3〜7/4のdev-client大量テストと概ね一致
   - 仮説: dev-client（App Checkデバッグプロバイダ）の**デバッグトークンがFirebase Consoleに未登録**のため、開発ビルドのリクエストが全部「無効」判定になっている。TestFlight（App Attest）分は確認済みに計上されており本番経路は正常
   - 影響: 現状は enforcement 未有効化のため実害なし。ただし**この状態で enforcement を有効化すると dev-client が全滅する**ため、有効化前に ①dev-clientのデバッグトークン登録（起動ログに出るトークンを Console → App Check → アプリ → デバッグトークンの管理 に追加） ②登録後に無効リクエストが消えることの確認、が必須。release-checklist の enforcement 有効化前チェックに直結

6. **Firestoreに過去テストのデータが大量に残存（削除・クリーンアップの構造的漏れ）**
   - 観測（2026-07-04、Firebase Console）:
     - `households`: 10件以上のドキュメントが残存（現役はテスト用1世帯のみのはず）
     - `inviteCodes`: 旧6文字・失効済みを含む20件超が堆積
     - `users`: Authユーザー2人に対しドキュメント6件（孤児4件）
   - 構造的な原因の候補（修正フェーズで棚卸し）:
     - 「認証解除と全データ削除」が `inviteCodes` を削除対象に含めていない（最後の1人退出フローには含まれるのに非対称）
     - 招待コード再発行時、旧コードは `disabledAt` を立てるのみで**ドキュメント自体は削除されない**ため無限に堆積する（PLANのTicket 22は「古い /inviteCodes/{code} を削除する」と記載しており実装と不一致の可能性）
     - #2/#3 の削除フロー失敗・中断により、削除しきれなかった世帯・usersが積み残る
     - 過去ビルドのテストで削除フローを通さず放置した世帯も混在しており、全部がバグ由来とは限らない
   - 対応方針（記録のみ）: ①削除フロー修正（#2/#3）とあわせて削除対象の完全性（inviteCodes含む）を見直す ②再発行時の旧コード削除 or 定期クリーンアップの方針決定 ③本番リリース前に手動で一度全クリーンアップ（テストデータ一掃）を実施する
   - 補足: `households` 内の残存世帯にゴースト（親ドキュメント削除済み・サブコレクション残り＝Console上で斜体表示）が含まれるかは未確認。クリーンアップ時に確認する

7. **IAPゲートがproductionビルドで無効（購入アラートが出ない）**
   - 再現手順: build 26（`eas.json` production profile に `EXPO_PUBLIC_CSV_IMPORT_IAP_ENABLED=1` 設定済み）で、未購入端末からCSV取り込みをタップ
   - 実際の結果: 購入アラートが表示されず、ファイル選択画面へ直行（本人端末・未購入の家族端末の両方で再現＝購入キャッシュ由来ではない）
   - 期待結果: 購入アラート（¥300で購入/購入を復元/キャンセル）が表示される
   - 有力仮説: Expoの `EXPO_PUBLIC_*` 環境変数は**ソース中のリテラルな `process.env.EXPO_PUBLIC_X` 式のみビルド時にインライン置換**される。`lib/csvImportPurchaseGate.ts` は `process.env` をオブジェクトとして受け取り `env.EXPO_PUBLIC_CSV_IMPORT_IAP_ENABLED` と**間接参照**しているため、本番バンドルでは undefined になりゲート無効化。開発時はMetroが `process.env` に実値を注入するためdev-clientでは正常動作し、検出できなかった
   - 修正方針（未着手）: ゲート判定のenv読み取りをリテラル参照（`process.env.EXPO_PUBLIC_CSV_IMPORT_IAP_ENABLED === "1"` を直接書く）に変更し、テスト可能性はリテラル読み取り結果を注入する形で維持する。修正後は**TestFlightビルドでの再検証が必須**（この不具合はdev-clientでは再現しない）
   - 影響: build 26 を万一このままApp Store審査に出すと「IAPを添付したのに購入導線が存在しない」状態になりリジェクト濃厚。**審査提出は#7修正後のビルドで行うこと**

#### 参考

- CSVインポートIAP（expo-iap連携、2026-07-03実装）の購入・復元確認は、App Store Connectのプロダクト作成後に `EXPO_PUBLIC_CSV_IMPORT_IAP_ENABLED=1` ビルドで実施する（[docs/release-checklist.md](release-checklist.md) 参照）。ゲート無効のビルドではCSV取り込み挙動は従来どおり

- 実際の結果: 変更後に他端末で計画タブを開き直しても変更が反映されない
- 原因: 計画タブの保存が `get` / `set` のみで、`onSnapshot` による購読が未実装（build 14時点で既知）
- 暫定対応: なし。build 21にはonSnapshot実装を含めない
- 次アクション: その後 commit `63a4ee7` で計画タブ機能ごと廃止したため対応不要。詳細は [docs/decisions/plan-feature-retirement.md](decisions/plan-feature-retirement.md) を参照
