# Firestore 読み取り見積もりとインデックス方針

最終更新日: 2026-07-04

## 目的

App Store配布前に、主要画面でのFirestore読み取り量の目安を揃え、必要なインデックス・取得範囲制限の方針を明確化する。

## 前提

- 単位は「1世帯あたり」の概算
- `useCollection` の `onSnapshot` は初回に該当ドキュメントを読み、以後は更新差分のみ受信
- 履歴リストは全件リアルタイム購読ではなく、`limit(100)` のカーソルページングを使う
- 履歴カレンダーと集計タブの取引読み取りは、Firestoreディスクキャッシュ + `households/{id}/meta/dataVersion` マーカーでフレッシュネスを判定し、変更がない場合はサーバーから取引を再読込しない
- 履歴検索のカテゴリ/内訳/お店候補、記録/履歴編集のお店候補は、`stores` ではなく Firestore SDK ローカルキャッシュ内の取引スナップショットから生成する。候補表示そのものはサーバーreadを発生させない
- データ件数記号:
  - `N`: 全取引件数
  - `M`: 対象月の取引件数
  - `Y`: 対象年の取引件数
  - `P`: 履歴リスト1ページの件数（現行100件）
  - `A`: 口座件数
  - `C`: カテゴリ件数
  - `B`: 内訳件数
  - `G`: 予算件数
  - `V`: データバージョン・マーカー読み取り（通常1件）

## 主要画面の読み取り見積もり

### 記録タブ

- 初回読込: `A + C(type別) + B`
- 保存時追加読込: 基本はなし（書き込み中心）
- 備考: リアルタイム購読中は差分更新のみ。お店候補はSDKローカルキャッシュ内の取引から生成し、`stores` は読まない

### 履歴タブ（リスト）

- 初回読込: `V + min(N, P)`（変更なし・キャッシュ有効時は `V` のみで先頭ページ再読込なし）
- 追加読込: スクロールごとに最大 `P` 件（`startAfter` + `limit(100)`）
- 検索適用時: `readAll` スコープで対象範囲を全件取得し、種別/カテゴリ/内訳/店舗/メモをクライアント側フィルタ。キャッシュ版が最新ならサーバー全件読みを避け、版不一致時は差分または対象範囲を再読込する
- 検索候補: カテゴリ/内訳/お店候補はSDKローカルキャッシュ内の取引から生成し、`listTransactions`（現在表示・検索対象の配列）だけには依存しない
- 備考: 全件購読は廃止済み。リアルタイム反映ではなく、画面フォーカス/プルリフレッシュ/ミューテーション後の再読込で更新する

### 履歴タブ（カレンダー）

- 初回読込: `V + M`（変更なし・キャッシュ有効時は `V` のみで月内取引再読込なし）
- 月変更時: 対象月スコープのキャッシュが古い場合のみ `M` を再読込

### 集計タブ

- 初回読込: `V + Y + G + C(expense)`（変更なし・キャッシュ有効時は `V + G + C(expense)` のみで年内取引再読込なし）
- 年変更時: 対象年スコープのキャッシュが古い場合のみ `Y` を再読込
- 備考: 年次取引はリアルタイム購読せず、キャッシュとマーカー判定で読み取りを抑制する

### 設定タブ

- 初回読込: マスタ系（カテゴリ/内訳/口座/世帯メンバー等）
- CSV出力時: `N`（全取引を取得して整形）

## 現行クエリのインデックス方針

現行実装（`lib/firestore.ts`, `app/(tabs)/history.tsx`, `app/(tabs)/summary.tsx`）を確認した範囲では、複合インデックス必須のクエリは運用上ほぼ発生しない。

### 現行で使っている代表クエリ

- `transactions.where("date", ">=", from).where("date", "<=", to).orderBy("date", "desc")`
- `transactions.where("date", ">=", yStart).where("date", "<=", yEnd)`
- `transactions.orderBy("date", "desc")`
- `categories.where("type", "==", type)`
- `transactions.where("accountId", "==", id)` / `where("categoryId", "==", id)` / `where("breakdownId", "==", id)`

`stores.where("name", "==", trimmed)` は旧ビルド/旧データ互換関数に残るが、通常の記録・履歴編集・CSV取り込み・履歴検索候補では使わない。

## 取得範囲制限・ページング方針

- 履歴リストは `limit(100)` + `startAfter` のカーソルページングを採用済み
- リスト描画は `FlatList` で仮想化し、初回100件 + スクロール追加100件に制限する
- 履歴検索/集計ドリルダウンは、日付条件のみサーバー側 where で絞り込み、範囲内データへクライアント側フィルタを適用する。検索条件だけを編集している間は読み取りを発生させず、「検索」押下時に適用済み条件へ反映する
- 履歴カレンダーと集計タブは、取引読み取りをライブ購読せず、Firestoreディスクキャッシュ + `dataVersion` マーカーで必要時だけサーバー再読込する
- 将来 `N` や `Y` がさらに増えて体感遅延や課金増が見えた場合は、以下を追加検討する
  - 年別/月別など、より細かいデータバージョン・マーカー
  - 日付以外の検索条件をサーバー側へ寄せるための複合インデックス
  - CSV出力や年次集計の分割取得/バックグラウンド処理

## 月次原価の試算式（1世帯あたり）

本ドキュメントの読み取り見積もり（`N`,`M`,`Y`,`A`,`B`,`C`,`G`）を使って、操作量を月次概算する。

- 読み取り概算（例）
  - 記録タブ初回: `A + C + B`
  - 履歴リスト初回: `V + min(N, P)`（キャッシュ有効かつ変更なしなら `V`）
  - 履歴カレンダー初回: `V + M`（キャッシュ有効かつ変更なしなら `V`）
  - 集計タブ初回: `V + Y + G + C`（キャッシュ有効かつ変更なしなら `V + G + C`）
  - CSV出力1回: `N`
- 書き込み概算（例）
  - 取引追加1件あたり、`transactions` + `accounts` 更新 + `meta/dataVersion` 更新
  - 通常導線では `stores` / `storeCategoryUsage` を更新しない（旧ビルド/旧データ互換関数のみ残存）
  - 取引更新/削除はトランザクション内更新回数に比例

金額換算は次の式で行う。

- `月次コスト ≒ 読み取り回数 × (read単価/100,000) + 書き込み回数 × (write単価/100,000) + 削除回数 × (delete単価/100,000) + ストレージGiB × storage単価 + 送信GiB × egress単価`

注:

- 単価はロケーション・通貨で変わるため、最終値は Cloud Platform SKU を正とする。
- Firestore Security Rules評価で追加read課金が発生し得るため、見積もりにはバッファを持たせる。

### iOS オフラインキャッシュの効果

React Native Firebase iOS は **オフライン永続化がデフォルト有効**。同じセッション内・アプリ再起動時の購読初期化は、まずローカルキャッシュから読み出され、その後サーバから差分のみ取得する。**キャッシュ読み取りは課金対象外**。

そのため実測の課金対象read数は naive な「`N` × 起動回数」よりも大幅に少なく、初回インストールや長期離脱後のキャッシュ無効化時のみ全件読み取りが発生する。

### プロファイル別の月次読み取り見積もり

「保守的（毎回サーバーから履歴先頭ページを読む）」と「実勢（キャッシュ + `dataVersion` マーカーで変更なし時の再読込を抑制）」の2系列で見る。旧実装のような `N × 起動回数` の全件購読は行わない。

| プロファイル | 保守的: 月次read数            | 実勢: 月次read数             | 月次write数 |
| ------------ | ----------------------------- | ---------------------------- | ----------- |
| ライト       | 30 × (100 + 50 + 1) = 4,530   | ~500〜1,500（cache hit前提） | ~40         |
| ミディアム   | 240 × (100 + 50 + 1) = 36,240 | ~3,000〜15,000               | ~120        |
| ヘビー       | 480 × (100 + 50 + 1) = 72,480 | ~10,000〜50,000              | ~250        |

> 補足: 保守的値は「起動ごとに履歴先頭100件 + マスタ約50件 + マーカー1件をサーバー読み」とした粗い上限。集計年次やCSV出力を頻繁に使う場合は `Y` / `N` 分が追加される。実勢値は「変更なし時はマーカー1件のみで取引再読込なし」を仮定。実機テレメトリ取得後に再校正する。

### 価格設定への含意

Firestoreコストは「想定よりかなり安い」レンジに収まる見込み。価格決定の主要因は **Firestore原価ではなく Apple手数料・サポート負荷・他競合価格** になる。

#### Apple App Store 手数料

- サブスク: 初年度 30%、2年目以降 15%
- 買い切り: 30% （Small Business Program 適用なら 15%）

#### Apple Developer Program 年会費の影響

固定費 ¥12,800/年（USD $99 × ~¥130）を回収するために必要な有料ユーザー数:

- サブスク ¥240/月: **5〜6人**で年会費回収
- 買い切り ¥1,800: **9〜11人**で年会費回収
- 100世帯有料化で年商 ¥288,000（月額）〜 ¥240,000（年額）

#### 初期リリースの採用方針

Firestore原価は低いため、初期リリースでは継続コスト回収を主目的にしたサブスクは採用しない。ユーザー自身の家計データを取り出す導線をロックしないため、CSV出力は無料のまま残し、CSVインポートのみ非消耗型IAP（買い切り）でロック解除する。

1. **基本機能は無料**: 記録追加、同期、閲覧、集計、CSV出力は未購入でも利用可能にする
2. **CSVインポートのみ買い切りIAP**: データ移行・復元・大量投入の便利機能として課金する
3. **課金単位はApple ID単位**: 世帯単位課金やサブスクは初期リリースでは採用しない
4. **購入状態とFirestore利用権限は分離**: 未購入でもデータ閲覧、CSV出力、認証解除と全データ削除は制限しない

> 初期リリースのCSVインポート買い切り価格は ¥300 とする。競合家計簿アプリのCSVインポート/データ移行機能の有料・無料範囲は、将来の価格改定や機能追加時に再調査する。

## 予算アラート設定ランブック

Google Cloud Console の [お支払い] > [予算とアラート] で設定する。

1. 範囲: 本番請求先アカウント（または本番プロジェクト）
2. 期間: 月次
3. 予算額: まずは小さめ固定額（例: 想定月次コストの 2〜3倍）
4. しきい値: 50% / 90% / 100%（必要なら 110% 予測も追加）
5. 通知: 請求管理者 + 運用連絡先メール

運用ルール:

- アラート受信時は、履歴リストのページングが想定どおり `limit(100)` に収まっているか、日付範囲検索/CSV出力/集計年次読込（`Y` / `N`依存）が過剰に発生していないかを優先点検する。
- 予算は「自動停止」ではないため、必要時は手動で機能制限や調査を実施する。
