# Firestore 読み取り見積もりとインデックス方針

最終更新日: 2026-05-10

## 目的

App Store配布前に、主要画面でのFirestore読み取り量の目安を揃え、必要なインデックス・取得範囲制限の方針を明確化する。

## 前提

- 単位は「1世帯あたり」の概算
- `useCollection` の `onSnapshot` は初回に該当ドキュメントを読み、以後は更新差分のみ受信
- データ件数記号:
  - `N`: 全取引件数
  - `M`: 対象月の取引件数
  - `Y`: 対象年の取引件数
  - `A`: 口座件数
  - `C`: カテゴリ件数
  - `B`: 内訳件数
  - `G`: 予算件数

## 主要画面の読み取り見積もり

### 記録タブ

- 初回読込: `A + C(type別) + B`
- 保存時追加読込: 基本はなし（書き込み中心）
- 備考: リアルタイム購読中は差分更新のみ

### 履歴タブ（リスト）

- 初回読込: `N`（全取引を `date desc` で購読）
- 検索時: 追加読込なし（クライアント側フィルタ）
- 備考: 取引件数増加時の主要ボトルネック

### 履歴タブ（カレンダー）

- 初回読込: `M`（月範囲クエリ）
- 月変更時: 変更後の `M` を再読込

### 集計タブ

- 初回読込: `Y + G + C(expense)`
- 年変更時: `Y` を再読込
- 備考: 年次クエリは件数が大きくなりやすい

### 設定タブ

- 初回読込: マスタ系（カテゴリ/内訳/口座/世帯メンバー等）
- CSV出力時: `N`（全取引を取得して整形）

## 現行クエリのインデックス方針

現行実装（`lib/firestore.ts`, `app/(tabs)/history.tsx`, `app/(tabs)/summary.tsx`）を確認した範囲では、複合インデックス必須のクエリは運用上ほぼ発生しない。

### 現行で使っている代表クエリ

- `transactions.where("date", ">=", from).where("date", "<=", to).orderBy("date", "desc")`
- `transactions.where("date", ">=", yStart).where("date", "<=", yEnd)`
- `transactions.orderBy("date", "desc")`
- `categories.where("type", "==", type)`
- `stores.where("name", "==", trimmed).limit(1)`
- `transactions.where("accountId", "==", id)` / `where("categoryId", "==", id)` / `where("breakdownId", "==", id)`

## 取得範囲制限・ページング方針

- 現行は履歴リストで全件購読（`N`件）を採用
- 目安として `N` が増加して体感遅延や課金増が見えたら、以下を優先導入する
  - 履歴リストのページング（`limit` + カーソル）
  - 初期表示範囲の月制限（例: 直近6〜12か月）
  - 検索時のサーバー側絞り込み導入（必要条件に限定）

## 料金・無料枠の確認結果（2026-05-10）

公式ドキュメントで以下を確認済み。

- Firestore無料枠（1プロジェクトあたり無料DBは1つ）
  - 保存データ: 1 GiB
  - 読み取り: 50,000/日
  - 書き込み: 20,000/日
  - 削除: 20,000/日
  - 送信データ転送: 10 GiB/月
- 課金対象は、読み取り/書き込み/削除、ストレージ、ネットワーク帯域。
- 予算アラートはGoogle Cloud Billingの予算機能で設定する。

参照:

- [Firestore 料金](https://cloud.google.com/firestore/pricing?hl=ja)
- [予算と予算アラート（Cloud Billing）](https://docs.cloud.google.com/billing/docs/how-to/budgets)

## 月次原価の試算式（1世帯あたり）

本ドキュメントの読み取り見積もり（`N`,`M`,`Y`,`A`,`B`,`C`,`G`）を使って、操作量を月次概算する。

- 読み取り概算（例）
  - 記録タブ初回: `A + C + B`
  - 履歴リスト初回: `N`
  - 履歴カレンダー初回: `M`
  - 集計タブ初回: `Y + G + C`
  - CSV出力1回: `N`
- 書き込み概算（例）
  - 取引追加1件あたり、`transactions` + `accounts` 更新 + 必要に応じた `stores` / `storeCategoryUsage`
  - 取引更新/削除はトランザクション内更新回数に比例

金額換算は次の式で行う。

- `月次コスト ≒ 読み取り回数 × (read単価/100,000) + 書き込み回数 × (write単価/100,000) + 削除回数 × (delete単価/100,000) + ストレージGiB × storage単価 + 送信GiB × egress単価`

注:

- 単価はロケーション・通貨で変わるため、最終値は Cloud Platform SKU を正とする。
- Firestore Security Rules評価で追加read課金が発生し得るため、見積もりにはバッファを持たせる。

## 予算アラート設定ランブック

Google Cloud Console の [お支払い] > [予算とアラート] で設定する。

1. 範囲: 本番請求先アカウント（または本番プロジェクト）
2. 期間: 月次
3. 予算額: まずは小さめ固定額（例: 想定月次コストの 2〜3倍）
4. しきい値: 50% / 90% / 100%（必要なら 110% 予測も追加）
5. 通知: 請求管理者 + 運用連絡先メール

運用ルール:

- アラート受信時は、履歴リスト全件購読（`N`依存）と集計年次読込（`Y`依存）を優先点検する。
- 予算は「自動停止」ではないため、必要時は手動で機能制限や調査を実施する。
