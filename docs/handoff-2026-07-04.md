### #2/#3/#6 の根本原因（未コミット修正の中身）

原因は2つ:

1. **削除リストに `members` が含まれていた**（`lib/accountDeletion.ts`）。membersを先に消すと
   Security Rulesの `activeMember` 判定（自分のmemberドキュメントの存在が条件）が偽になり、
   以降の削除がすべて permission-denied になる
2. **退出フローの inviteCodes 検索が `allow list: if false` で拒否**されていた

対応（変更ファイル）:

- `firestore.rules`: inviteCodes の list を「自世帯のコードのみ」許可
  （`activeMember(resource.data.householdId)` 条件。membersサブコレクションが根拠のため
  users.householdId を偽造した非メンバーは突破できず、総当たり耐性は維持。
  経緯は [docs/decisions/invite-join-bruteforce-mitigation.md](decisions/invite-join-bruteforce-mitigation.md) に追記済み）
- `lib/accountDeletion.ts`: 削除リストから `members` を除外、`meta` / `joinRequests` を追加。
  契約テスト（`lib/accountDeletion.test.ts`）も新契約に更新済み
- `lib/firestore.ts`: 全データ削除を「データ → inviteCodes → (members+世帯docを1バッチ) → users → Auth」の
  順序に修正（1バッチ内はRulesがバッチ前状態で評価されるためmembersと世帯docを同時に消せる）
- `lib/household.ts`: 最後の1人退出フローを同じ順序に修正
- `firestore.rules.test.ts`: 回帰テスト6件追加（退出シーケンス成功、members先行削除のロックアウトガード等）

### #1 の対応（未コミット修正の中身）

仕様決定（ミノ）: **参加は承認制なので、退出/解除済みユーザーの再参加は承認フローに委ねて許可する。**

- `lib/household.ts`: `rejoinDisabled` フラグ廃止（書き込み・承認時チェックとも削除。過去データのフラグは無視）、
  自己退出時に自分の `users.householdId` を即時クリア（「すでに参加しています」誤判定の解消）、
  未使用のレガシー直接参加関数 `joinHousehold` を削除
- `ARCHITECTURE.md` / `docs/known-issues.md`（G1改訂）に方針変更を記録済み
